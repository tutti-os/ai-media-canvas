# Loomic 单机版实施基线

> 用途：这不是详细施工清单，而是长任务的执行基线。它回答 4 个问题：
>
> 1. 最终要交付什么  
> 2. 每一阶段做到什么才算过关  
> 3. 需要靠什么测试与验证来防跑偏  
> 4. 哪些决策必须先定，不能边做边飘

---

## 1. 基线目标

本项目最终要交付的是一个**本地 Web 单机版 Loomic**，满足以下硬约束：

- 无账号
- 无登录
- 无注册
- 无 auth callback
- 无 sign out
- 无 Supabase
- 无积分
- 无支付
- 无订阅
- SQLite 持久化
- 本地文件系统存储资产
- 打开即进入工作区
- 尽量保留现有视觉与交互体验

一句话定义：

- 最终成品必须看起来像一个“本地创作工具”，而不是“去掉了登录页的 SaaS”

---

## 2. 非目标

为了避免 scope 膨胀，以下内容不属于首版必须目标：

- 重做 UI 风格
- 顺手重构整个代码结构
- 引入桌面壳（Electron/Tauri）
- 优先追求最漂亮的数据建模
- 先做技能市场、社区发现、在线字体等联网增强能力

首版更重要的是：

- 跑通本地工作流
- 去掉云依赖
- 不漏核心功能
- 不把样式改歪

---

## 3. 最终交付定义

当且仅当以下条件同时成立，才算单机版完成：

### 3.1 产品形态

- 打开应用后直接进入最近项目或默认项目
- 没有 `/login`
- 没有 `/register`
- 没有 `/auth/callback`
- 没有 `/pricing`
- 没有 `Billing / Usage / Subscription` 语义

### 3.2 运行时

- 不再要求任何 `SUPABASE_*` 环境变量
- 不再要求任何 `LEMONSQUEEZY_*` 环境变量
- 不再依赖 `@supabase/supabase-js`
- 不再依赖 Supabase Storage
- 不再依赖 PGMQ

### 3.3 本地能力

- 能本地创建项目
- 能本地保存画布
- 能本地保存聊天记录
- 能本地保存上传文件与生成结果
- 重启应用后数据还在

### 3.4 语义收口

- 不再暴露 `viewer / workspace / member / billing / subscription` 作为最终产品语义
- 允许作为迁移兼容层短期存在，但最终交付前必须清理掉

### 3.5 首版页面与功能边界

首版默认保住这些页面或能力：

- `projects`
- `canvas`
- `brand-kit`
- `settings`
- 项目创建/删除/进入
- 画布保存/恢复
- 聊天会话
- 上传与本地资源回读
- 至少 1 条图片生成主路径

首版默认不作为核心交付：

- 营销首页
- `login / register / auth callback`
- `pricing / billing / credits / subscription`
- `home` 发现流
- `skills` marketplace / 在线安装
- 在线字体发现能力

如果后续决定保留 `home` 或 `skills`，必须被视为“第二层增强能力”，不能阻塞本地核心工作流上线。

## 4. 先决决策

这些决策不一定要在第一天全部做完，但必须在对应阶段开工前定死。

### 决策 A：工作区 IA

要选哪一种：

1. 保留多页面工作区
   - `projects / brand-kit / settings`
   - `home / skills` 视情况保留
2. 收缩为单工作区
   - 以 `project/canvas` 为核心

当前建议：

- 首版先保留 `projects / brand-kit / settings`
- `home / skills` 单独决策，不默认纳入核心

### 决策 B：生成链路

要选哪一种：

1. 图片同步、视频可暂缓
2. 图片同步、视频本地 job
3. 保留完整 job / worker 模型

当前建议：

- 首版先保图片生成同步可用
- 视频生成不作为第一优先级

### 决策 C：Agent 持久化

要选哪一种：

1. 保留可恢复 agent
2. 明确降级为非持久化 agent

当前建议：

- 如果目标是尽快交付单机版，首版允许降级
- 但必须在产品层明说，不能做成“看起来支持、实际不稳定”

### 决策 D：联网增强能力

包括：

- Home 发现内容
- Skills marketplace / URL import
- Google Fonts 发现与预览

当前建议：

- 不纳入首版核心验收
- 若保留，明确标为“联网增强能力”

## 5. 阶段基线

这里的“阶段”不是详细任务，而是执行时必须逐阶段验收的关卡。

### 阶段 0：副本冻结

目标：

- 复制 `Loomic` 到 `Loomic-standalone`
- 副本能独立启动
- 当前视觉、页面、交互与原项目一致

过关标准：

- `Loomic-standalone` 可以独立运行
- 不污染原始仓库
- 能访问当前页面与工作区

验证方式：

- 手工启动验证
- 页面打开验证

### 阶段 1：入口收口

目标：

- 去掉营销首页和定价入口
- 入口开始向工作区收口
- 暂时允许保留 auth 占位链路，避免工作区死跳转

过关标准：

- `/` 不再是营销漏斗
- `pricing` 不再是主入口
- 工作区仍可进入

验证方式：

- 手工访问 `/`
- 检查是否还能进入工作区

### 阶段 2：本地身份兼容层

目标：

- 让后端先脱离真实 Supabase 登录态
- 为前端去 token 做准备

过关标准：

- 在无真实 Supabase 登录态下，核心接口仍能返回有效上下文
- 工作区、项目、设置、生成链路至少可以继续跑核心路径

验证方式：

- 集成验证 `/api/viewer` 或等价 bootstrap
- 项目列表 / 项目创建 / 画布读取基本可用

注意：

- 这是迁移阶段兼容层，不是最终成品形态

### 阶段 3：前端去 auth

目标：

- 去掉 `useAuth()` 主依赖
- 去掉 `session/access_token` 控制流
- 去掉 `/login` 跳转心智

过关标准：

- 前端不再以 `session?.access_token` 为前置条件
- 不再有 `ApiAuthError -> signOut -> /login`
- `useCreateProject`、`useDeleteProject`、`useWebSocket` 可在无登录前提下工作

验证方式：

- 手工创建项目
- 手工删除项目
- 打开工作区不跳 `/login`
- WebSocket 建连可用

### 阶段 4：SQLite 最小兼容底座

目标：

- 建立本地 SQLite 持久化
- 建立最小兼容表结构与事务语义

过关标准：

- 应用冷启动可自动生成或迁移本地数据库
- 默认项目/默认画布可创建
- 基础业务数据可读写

必须保留的兼容语义：

- `create_project + primary canvas`
- `bootstrap_viewer` 的本地等价逻辑
- `chat_sessions.thread_id`（若保留可恢复 agent）
- `chat_messages.content_blocks`
- `projects.thumbnail_path`
- `brand_kit_assets`

验证方式：

- 项目创建后自动有 primary canvas
- 保存后重启仍能读回

### 阶段 5：核心业务链路迁移

目标：

- 先保住最关键工作流

优先级：

1. projects
2. canvases
3. chat
4. settings
5. uploads
6. brand kit

过关标准：

- 项目可创建/编辑/删除
- 画布可保存/恢复
- 聊天会话可创建/切换/保留
- 设置可保存

验证方式：

- 集成测试 + 手工工作流测试

### 阶段 6：本地文件存储

目标：

- 所有关键资产走本地磁盘

覆盖范围：

- project thumbnail
- canvas files
- brand kit assets
- generated image/video assets
- agent runtime persistImage

过关标准：

- 上传与生成结果在重启后仍可访问
- 不再依赖 Supabase Storage

验证方式：

- 文件路径检查
- 重启后资源回读验证

### 阶段 7：生成链路落地

目标：

- 明确并实现首版保留的生成路径

过关标准：

- 至少 1 条图片生成主路径可用
- 插图回写画布可用
- 若保留长任务，job 状态反馈可用

验证方式：

- 端到端生成测试
- 画布插入验证

### 阶段 8：清理付费与云端残留

目标：

- 去掉积分、支付、定价、套餐限制等所有 SaaS 残留

过关标准：

- 不再出现 credits / billing / pricing / subscription 语义
- 不再有 `TierLimitToastProvider`、`CreditHeaderButton`、`CreditBalance` 等入口

验证方式：

- UI 手工扫描
- 文案 grep

### 阶段 9：最终路由与体验收口

目标：

- 变成真正的本地创作工具入口体验

过关标准：

- 打开即进入项目
- 默认项目/最近项目自动打开
- 无登录页、无首页、无定价页

验证方式：

- 冷启动手工验证

### 阶段 10：最终去 auth / 去云端语义收尾

目标：

- 删除所有迁移兼容层残留
- 保证最终成品没有 auth / viewer / workspace / Supabase 心智

过关标准：

- 不再存在 `RequestAuthenticator`
- 不再存在 `AuthenticatedUser.accessToken/email/userMetadata` 作为运行时前提
- 不再存在 WS `token` 查询参数
- 不再存在 `RunCreateRequest.accessToken`
- 不再存在 `viewer.workspace.id` 作为前端必需语义
- 不再依赖 `@supabase/supabase-js`

验证方式：

- 代码 grep
- 启动验证
- 端到端工作流验证

## 6. 测试基线

这类改造不能只靠单测，也不能只靠手工点点看。建议用 3 层验证。

### 6.1 单测

优先覆盖这些：

- SQLite repository
- 本地文件存储 helper
- 数据转换逻辑
- marker/resolver 协议
- 路由参数解析
- bootstrap / default project 逻辑

### 6.2 集成测试

至少覆盖这些链路：

- 创建项目会同时创建 primary canvas
- 读取/保存 canvas
- 创建/读取聊天消息
- 更新设置
- 上传文件并回读
- 本地 bootstrap

### 6.3 手工工作流验收

这是首版最关键的部分，至少每个阶段都要做。

必须手工验证：

- 冷启动是否能进入工作区
- 项目创建是否正常
- 画布保存后刷新是否正常
- 重启应用后数据是否还在
- 聊天侧边栏是否还能用
- 上传和生成结果是否还能回显
- 不再出现 `/login`
- 不再出现 pricing / credits / billing

---

## 7. 首版必须通过的验收用例

下面这些不是可选项，是首版最低可交付门槛。

### 用例 1：首次启动

步骤：

1. 删除本地数据库和 assets 目录
2. 启动应用
3. 打开本地地址

期望：

- 应用能自动初始化
- 自动创建默认项目/默认画布或自动进入项目选择
- 不出现登录页

### 用例 2：项目创建

步骤：

1. 在 UI 中创建项目
2. 进入项目画布

期望：

- 项目创建成功
- 自动拥有 primary canvas
- URL 能正常进入工作区

### 用例 3：画布保存与恢复

步骤：

1. 在画布插入元素
2. 保存或触发自动保存
3. 刷新页面
4. 重启应用再打开

期望：

- 刷新后内容还在
- 重启后内容还在

### 用例 4：聊天会话

步骤：

1. 打开 ChatSidebar
2. 创建会话
3. 发送消息
4. 切换会话再切回

期望：

- 消息保留
- 会话切换正常

### 用例 5：上传与本地文件

步骤：

1. 上传图片
2. 在画布或品牌资产中使用
3. 重启应用再打开

期望：

- 文件落到本地
- 重启后仍能访问

### 用例 6：图片生成

步骤：

1. 发起一次图片生成
2. 等待结果
3. 检查是否插入画布

期望：

- 至少一条图片生成主路径可用
- 结果可持久化

### 用例 7：SaaS 残留清理

步骤：

1. 浏览主页面
2. 浏览设置页
3. 搜索 UI 文案与路由

期望：

- 没有 login / register / auth callback
- 没有 pricing / credits / billing / subscription
- 没有 sign out

## 8. 不允许回归的红线

以下任意一条出现，都说明实现已经偏离基线：

- 改完后仍要求 Supabase 环境变量
- 改完后仍要求登录
- 改完后主工作区样式明显跑偏
- 改完后项目/画布不能持久化
- 改完后上传/生成结果不落本地
- 改完后还保留明显的 SaaS 语义
- 改完后只“本地能跑”，但兼容层残留成最终形态

---

## 9. 推荐执行节奏

推荐按下面这个节奏推进，而不是一口气长改：

1. 先过阶段 0 到阶段 2
2. 确认入口、身份兼容、前端去 auth 没问题
3. 再做 SQLite 与核心业务链路
4. 再做文件存储和生成
5. 最后统一做去 SaaS 语义收尾

每完成一个阶段，都至少做：

- 1 次手工工作流验证
- 1 次关键链路测试
- 1 次“是否还残留 Supabase / auth / billing 语义”的扫描

---

## 10. 这份基线怎么用

这份文档不是用来代替详细施工计划的，而是用来约束施工方向。

执行时应遵循：

- 每一个阶段都先看“过关标准”
- 每一次改动都对照“首版必须通过的验收用例”
- 每一次阶段收尾都检查“红线是否被触发”

如果后面要继续往下拆，建议下一份文档再写：

- `阶段任务拆解.md`

那时再按阶段把文件级动作和实现任务逐条展开。
